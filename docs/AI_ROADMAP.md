> 基于当前已完成的阶段一（静态启发式策略 + GE 评分 + 团队视角 + 特殊建筑联动），后续分阶段增强 AI 决策能力。

## 0. 评估体系（贯穿全路线图）

### 0.1 问题：旧评估的局限

之前的评估只看胜率/平均分/城市大小，但 **AB 队使用同一套策略** 时，胜率永远是 ~50%，看不出策略本身的好坏。

### 0.2 双轨评估策略

#### 轨 A：纵向绝对指标（核心）

每步决策时，枚举该步所有合法 move，**对每个 move 用当前策略的评分函数打分**，看实际选中的 move 排名：

| 指标 | 含义 | 目标值 |
|------|------|--------|
| 选角 Top-1 命中率 | 选的角色的评分是否等于最高分 | ≥ 95% |
| 资源 Top-1 命中率 | 拿金/抽牌决策是否正确 | ≥ 90% |
| 建造 Top-1 命中率 | 建的牌的 buildScore 是否最高 | ≥ 85% |
| 刺杀 Top-1 命中率 | 刺杀目标评分是否最高 | ≥ 70% |
| 偷窃 Top-1 命中率 | 偷窃目标评分是否最高 | ≥ 80% |
| 摧毁 Top-1 命中率 | 摧毁目标评分是否最高 | ≥ 85% |
| **综合质量指数** | 全部决策的加权平均 | ≥ 85% |

这不需要另一套 AI，纵向对比即可衡量进步。

#### 轨 B：横向对比

存档旧版 `AutoplayPolicy.ts`，切换新版后跑同一组随机种子评估，对比：

- 终局指标：平均分、城市大小、完成率
- 过程指标：铁匠铺/实验室/墓地使用率、刺杀/摧毁频次分布
- 稳定性：极端比分占比

### 阶段二（📋 当前计划）：静态策略增强 + 概率推理

#### 2.1 `predictLikelyRoles` 升级：排除法推理（预估 1 天）

**现状**：`predictLikelyRoles` 仅根据对手的城市颜色和手牌预测角色，没有利用天绝弃牌和已选角色池信息。

**目标**：与人类推理一致——根据"哪些角色已经消失（天绝 + 已选走）"来缩小对手可能角色的范围。

**实现方式**：

```
功能：已知消失角色集合（天绝 1 张 + 已选走 N 张），推断某对手可能选的角色
流程：
    1. 消失集合 = 天绝弃牌(未知是哪张，但首发玩家知道) + 已分配(公开的)的角色
    2. 候选角色 = 全部8张 - 已分配的角色（对首发玩家：- 天绝弃牌）
    3. 对候选角色逐一打分（与 scoreCharacterPick 逻辑一致）
    4. 按分数排序，输出 Top-3 作为最可能的选中角色
    5. 刺客/盗贼从 Top-3 排序中选择目标
```

**基于游戏轮次的信息量递减**：
- 首轮选角结束后，场上信息最少（只有首发知道天绝弃了什么、自己选了啥）
- 第 2+ 轮开始后，角色归属信息逐渐暴露（被刺、被偷、公开的角色会越来越多）
- 服务端的 `isRolePubliclyKnown`（`CharacterManager.ts`）已经区分了公开/未知角色——AI 应只使用和人类一样的信息

**预期效果**：
- 刺杀概率命中率从当前 ~45% 提升至 ~60%
- 刺杀高威胁角色（军阀/建筑师/商人）的比例提高
- 减少刺杀到低价值角色（国王/主教）的概率

#### 2.2 选角博弈评分树（预估 1-2 天）

**现状**：「首发必拿刺客」是硬编码——始终选刺客，不考虑场景。

**需求**：你现在已经验证了大多数情况下首发必拿刺客是对的，但**后期极端场景**（某个对手坐拥重资、城市接近完成）需要放掉刺客去拿军阀/建筑师。

**实现方式**：替换硬编码 `return assassinIdx` 为软评分：

```
scoreCharacterPick(ASSASSIN) = 
    基础分 5 
    + 保护队友（队友有大量资源 → +N）  
    + 阻止对手（对手濒临完成 → +N）  
    + 极端放弃检测：如果某个剩余角色的自评分数远超刺客的阻止分数  
      且自己的城市/资源足够利用该角色  
      → 刺客评分下调，最高收益角色评分上调
```

**验证**：跑 `aiEval` 看首轮刺客选中率从 100% 降到 ~90%（大多数正常局仍是刺客，极端局放弃刺客改选高收益角色）。

#### 2.3 团队策略深度（协同识别，预估 0.5 天）

**现状**：AI 只能被动保护（队友有金币→刺盗贼保护；队友有手牌→刺魔术士保护）。不能主动配合队友。

**新增**：
- **选角协同**：如果队友选了建筑师（P3），作为 P1 拿刺客保护他的回合
- **建造协同**：配合队友的城市颜色——如果队友缺某色，自己优先补那个颜色
- **分路意识**：首发（P1/P2）倾向保护/阻止角色，中位（P3/P4）倾向发展角色，末位（P5/P6）倾向经济/收尾角色

#### 2.4 特殊建筑联动策略细化（预估 0.5 天）

**现状**：
- 铁匠铺：有了就用 2 金抽 3 张（粗粒度）
- 实验室：手牌多时卖低价值牌（粗粒度）
- 图书馆+天文台、魔法学校无特殊策略

**细化**：
- **图书馆 + 天文台**（同时拥有）：抽牌 = 保留 3 张（极高价值），对应角色的选角评分 +N，资源的抽牌优先级提至高
- **魔法学校**：城里有时，收租收入角色的评分 +1/每个额外颜色。城市已经有 4 色时造魔法学校 = 立刻凑 5 色 → +6 分（建评+终局策略）
- **铁匠铺深策略**：如果手牌多但都是低费垃圾 + 有铁匠铺 → 拿金 + 铁匠铺抽牌（比直接抽牌更赚） vs 手牌多且质量高 → 直接拿金不抽

### 阶段三（🔭 中期计划）：蒙特卡洛树搜索

#### 3.1 动机

静态策略的极限——无论评分函数多精细，总有盲点。MCTS 通过自我对弈探索出静态策略不可见的战术。

#### 3.2 工程基础

**已有基础设施**：
- `TrainingEngine`：6 人 3v3 纯内存模拟，~70ms/局
- `aiEval.test.ts`：自动跑多局输出统计
- `pickAndApplyAutoplayMove`：完整的 AI 决策入口

**新增组件**（`server/src/engine/mcts/`）：
- `mctsTrain.ts` — MCTS 训练主循环
- `mctsNode.ts` — 搜索树节点（含 UCB 计算）
- `rolloutPolicy.ts` — Rollout 用的简化策略（比当前 AI 随机化，更快）

#### 3.3 搜索范围

《富饶之城》全状态空间太大无法穷举（牌库随机、隐藏信息）。MCTS 只用在**最高影响、最不确定**的单一决策点上：

**焦点决策**：选角阶段（每轮的选角决策决定了后续全部回合）。

**不适用 MCTS 的决策点**（保持静态策略）：
- 拿金/抽牌（2 选 1，策略简单）
- 建造（buildScore 已经很好）
- 刺杀/偷窃目标（食材不多，排除法足够）
- 摧毁目标（评分覆盖完整）

#### 3.4 算法设计

```
for each 选角决策点:
    1. 枚举当前剩余角色（N ≤ 6）
    2. 对每个角色, 创建 MCTS 根节点
    3. 有限次迭代（~200-500次）:
        a. 选择一个节点（UCB1）
        b. 展开一个子节点（模拟下一步选角或回合）
        c. 从该节点 Rollout（快速随机策略跑半局 = 3-4 轮）
        d. 反向传播结果（胜/负/平局分值）
    4. 每个角色的最终 score = visitCount 的访问量加权胜率
    5. 选 score 最高的角色作为当前策略的 "监督信号"
```

**Rollout 简化**：不完完整整局，跑 3-4 轮（约半局）就用中期优势（城市建设数分差）来近似终局胜率。中期优势与终局胜率高度相关。

**并行利用**：一个回合内的 6 个选角的 MCTS 是独立的，可以并行跑。

**时间估算**：一次决策点需要 ~200ms（~200 rollouts × ~1ms/rollout）。一局约 20 次选角 → 4 秒/局。50 局训练 → ~3 分钟。

#### 3.5 评估

训练完成后：
1. 用静态策略跑 100 局，记录决策质量
2. 用 MCTS 策略跑 100 局，记录决策质量
3. 对比选角 Top-1 命中率的提升（目标：从 ~90% → ~98%）
4. 对比与"静态策略"的 1v1（需用旧版策略文件替换验证）

### 阶段四（🔭 远期，无固定时间表）：神经网络

如果 MCTS 后的 AI 仍然可以进一步提升，考虑引入轻量神经网络：

```
模型：~50K 参数的小型 MLP 或小型 Transformer
输入：gameState 编码（~100 维：城市颜色数/金币/手牌数/角色池等）
输出：8 维（每个角色的选角胜率）+ 1 维（局面价值）
训练数据：来自 MCTS 的 ~100K 样本
```

**评估**：对比 MCTS 和神经网络在**推理速度**（ms/步）和**决策质量**上的 trade-off。

在 6 人 3v3 固定规则下，神经网络可能不是必需的——MCTS + 优良的静态策略已经足够覆盖人类玩家的战术水平。

### 第一条：P2 防守与"官刀"概念

> **二发防守责任大，通常必拿238，纵然身死无所谓，团队依旧笑哈哈。**

**解析**：P2（第二个选角位置）是 B 队的前置位，选牌非常关键。238 分别代表**盗贼(2)、魔术师(3)、军阀(8)**——三个攻击型角色。
拿这些角色一举两得：既可以攻击对手（偷钱/换牌/拆建筑），又可以防止对手拿了这些角色来攻击己方。
刺客(1)通常是首发玩家（P1，A 队前置位）拿的，P2 的主要职责不是拿刺客，而是拿 238。
即使 P2 被刺客砍中导致空过一轮也没关系——只要保护了后面的队友（P4/P6）安全发育，团队整体赚。

**算法映射**：
```
scoreCharacterPick(actor=P2):
    THIEF   评分 += 5  // P2 拿盗贼攻击对手+防止被偷
    MAGICIAN 评分 += 4  // P2 拿魔术师限制对手手牌
    WARLORD  评分 += 3  // P2 拿军阀拆对手关键建筑
    ASSASSIN 评分 -= 3  // 刺客留给首发(P1)，P2 不抢
    发展型角色(商人/建筑师) 评分 -= 3  // P2 不应贪发展
```

### 第二条：选角中的色房判断

> **手握蓝房常扣5，心存见合不杀8**

**解析**：手牌中有蓝色（宗教）建筑时，经常扣着选 5（主教），能收租又能防军阀。后半句指——如果你猜不透队友的意图（见合 = 局面不确定），不要轻易杀 8（军阀/官刀角色）。有时候队友是故意把军阀留给你去砍（保护你），有时候队友是想自己拿军阀拆对手房子。不确定的情况下宁愿不杀，即使对手拿到军阀拆了己方建筑（-1），也比队友拿到却被自己砍死（队友废一轮 + 对手无损失）强。

**算法映射**：
```
assassinTargets:
    if (角色归属未知):
        对该角色所属"是不是队友"的不确定性打分
        不确定性高 → 刺杀优先级下调
        // "心存见合不杀8" —— 不确定就别杀
        score *= (1 - uncertainty * 0.5)
```

> **我方无绿可砍6，杀7最易伤本家**

**解析**：我方队伍没有绿色（商业）建筑时，可以刺杀 6（商人）——因为商人收益来自商业区数量，我方无绿意味着商人拿了也没高收益。杀 7（建筑师）最容易砍到本家队友——因为建筑师是热门角色，队友很可能拿了。

**算法映射**：
```
scoreCharacterPick(actor):
    MERCHANT 评分 += 己方商业建筑数 × 2
    // 己方无绿 → 商人评分低 → 对敌人威胁小

assassinTargets:
    ARCHITECT 的"可能是队友"权重 +N
    // 建筑师太热门，队友/敌人都可能拿，杀7伤本家概率高
```

> **倒二有黄不杀4，敌有一点可偷8**

**解析**：倒数第二个选角位置（P5）有黄色（贵族）建筑时，不刺杀 4（国王）——国王的收益来自贵族区数量，P5 有黄说明自己拿国王收益高。对手有一点金币时（stash > 0），优先偷窃 8（军阀）——军阀需要金币启动摧毁能力，先偷走他的钱让他拆不起。

**算法映射**：
```
assassinTargets:
    if (己方有黄色建筑 && 4号国王还在池中):
        KING 的刺杀优先级 -= 6  // 自己有用，不刺

thiefTargets:
    if (对手.stash > 0):
        WARLORD 偷窃优先级 += (对手.stash + 城市大小) × 2
        MERCHANT 偷窃优先级 += 对手.stash × 1.5
```

> **37同甩威力大，五发队友把你夸**

**解析**：3（魔术师）和 7（建筑师）同一个人选（同甩），效果巨大。但如果 3 和 7 分开给两个人——四发拿了 3，五发只能拿 7 → 四发用魔术师攻击五发的手牌。所以三发的最佳策略是：3 和 7 都不拿，留给四发五发自己分配。四发拿 3（攻击五发），五发拿 7（补手牌），3 和 7 互相抵消。详细博弈流程见"三七同甩"示例。

**算法映射**：
```
scoreCharacterPick(actor=P3):
    if (池中有 MAGICIAN 和 ARCHITECT 且 P5队友手牌少):
        // 最佳策略：都不拿，留给四发五发
        // P4拿3攻击P5 → P5拿7补牌 → 互相抵消
        // 三发自己选另一张高收益角色
        MAGICIAN  评分 -= 3  // 留给四发
        ARCHITECT 评分 -= 3  // 留给五发
```

### 第三条："官刀"博弈的核心

> **贱命一条才可怕，四发就把官刀拿，我被刀砍同伴爽，砍我同伴我伤他**

**解析**：这是团队防守的最精华思想——"贱命一条才可怕"。

当四发选手（P4）前期发育不好、无法承担发展位时，主动转为进攻/防守位，拿"官刀角色"（当前局面下刀手最可能砍的角色）。这样：

| 场景 | 结果 |
|------|------|
| 刀手砍**官刀** → 砍中 P4 | P4 被牺牲，但队友（P3/P5）安全拿了高收益角色，团队赚 |
| 刀手**不砍**官刀 → P4 拿到官刀角色 | P4 拿着高风险高回报角色可以给对手造成极大伤害 |
| 刀手砍到 P4 的**其他队友** | P4 拿官刀角色成功发挥价值 |

**无论刀手怎么选，P4 都不亏**。——这就是"贱命一条才可怕"：价值不确定的角色（可能低价值，可能高价值）才是刺客最难处理的。

**算法映射**：
```
scoreCharacterPick(actor, seat=P4):
    if (actor 的城市和手牌都较少 —— "发育差"):
        // 转为防守位
        高风险高回报角色评分 += 5  // 拿"官刀角色"拼一把
        发展型角色评分 -= 4       // 已经没资本发展了

assassinTargets:
    // 预测"官刀"——当前刀手最可能砍的对象
    // 根据局势推演：如果我是刀手，我会砍谁？
    // 与 opponentAssassin 共享同一评分逻辑
    if (预测结果 == 实际选角结果):
        // 砍中了官刀
    else:
        // 刀手没砍官刀，可能是反向思维
```

### 第四条：同色截断策略

> **敌人色房成了片，我方防守必截他，工作总得有人干，谁有同色谁先拿**

**解析**：这是防守选牌策略。例如 P5 有三个绿色（商业）建筑 → 他拿商人的收益极大。那么 P2/P4 就必须截断商人，不让 P5 拿到。谁来截？——**谁手里也有绿色建筑**（即谁拿商人也有不错的收益）谁就来截。这样防守（阻止对手高收益）的同时还能给自己带来经济收益，一举两得。

**算法映射**：
```
scoreCharacterPick(actor, seat):
    // 扫描后续对手（尤其是紧邻的 1-2 个位置）的城市颜色
    foreach (后续对手):
        if (对手有大量同色建筑):
            对应收入角色评分大幅度上调 // 截断对手
            // 但仅限于己方也有该色建筑的人来截
            if (actor 也有该色建筑):
                评分 += 大量 // 一举两得
            else:
                评分 += 少量 // 纯防守

    // 示例：P2 看到 P5 有 3 个商业建筑
    // P2 若无绿色建筑 → 商人评分 +3（纯防守）
    // P2 若有绿色建筑 → 商人评分 +8（防守+收益）
```

### 第五条：三七同甩的连锁选角（核心团队博弈）

> **精细示例（实战积累）：**

六人 3v3，三发选牌时看到 3（魔术师）和 7（建筑师）都在池中：

- 如果五发队友手牌少（需要补牌），三发选了 7（建筑师）
  → 四发就会选 3（魔术师）去**攻击**五发——换走五发的手牌
  → 五发手牌本来就少，被换走更难补
  → 己方团队损失

- 如果三发选了 3（魔术师），自己没有额外收益（魔术师本身不产生资源）
  → 四发就可以拿到 7（建筑师）这个强牌
  → 对手四发受益，己方团队损失

**最优解**：三发 3 和 7 都不拿，选另一张高收益角色
→ 把 3 和 7 都留给四发和五发
→ 四发面对 3 和 7 只能选其一（假设选了 3 攻击五发）
→ 五发拿到剩下的 7（建筑师），立刻补手牌
→ 四发的攻击（3 换手牌）被五发的 7（抽牌补手牌）抵消
→ 三发自己还拿了高收益角色
→ **整个 135 团队的资源最大化**

**算法映射**：
```
scoreCharacterPick(actor, seat):
    // 连锁选角预测：模拟后续玩家的选角决策
    // 在当前回合，模拟后续 N 个玩家的选角：
    //   模拟 P4 根据池中剩余角色选角
    //   模拟 P5 根据池中剩余角色选角
    //   ...
    // 评估"如果我拿角色 X，后续会发生什么"的团队净收益

    simulateChainPick():
        netTeamScore = myScore(角色X)
        remainingPool = 当前池 - X
        for each 后续队友:
            模拟其最优选角(remainingPool)
            netTeamScore += 模拟得分
            remainingPool -= 模拟所选角色
        for each 后续对手:
            模拟其最优选角(remainingPool)
            netTeamScore -= 模拟得分 * 1.2  // 对手得分=己方损失
            remainingPool -= 模拟所选角色
        return netTeamScore
```

### 第八条：国王选角博弈

> **何时拿4有奥义，敌人数多放沉底，大局不是乱抢4，谁有黄房谁登基**

**解析**：

- 核心原则不是乱抢国王——**谁有黄色建筑（贵族区）多，谁就应该拿国王**才能收益最大化
- 如果自己没黄房还抢国王，等于白拿一个没经济价值的角色
