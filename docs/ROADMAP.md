- [DEPENDENCY_BASELINE.md](./DEPENDENCY_BASELINE.md) — 依赖策略

## P0 — 规格冻结（检查表）

> 目标：实现时不再争论规则；本 Phase 以文档与约定为主，可不写业务代码。

### 模式

- [x] **竞技**：仅 6 人；强制团队 135 vs 246（`playerOrder` 下标 0,2,4 vs 1,3,5）
- [x] **禁止** 6 人个人混战（防误开）
- [x] **娱乐**：非 6 人、或含 AI 等；可归档、**不进排名**
- [x] 娱乐规则不优先设计，沿用现有经典个人局逻辑

### 竞技计分

- [x] 建成城规模固定 **8**（`completeCitySize = 8`，竞技不可改）
- [x] 本局**第一个**建成（达 8 城）：个人 **+8**
- [x] **同一回合内**后续建成：个人 **+6**
- [x] 队伍总分 = 队内个人 `total` 之和
- [x] 队总分相同 → **平局**

### 账号

- [x] 注册/登录：帐号名 + 密码 + 游戏内昵称（默认同帐号名，可改）
- [x] 战绩榜显示**昵称**
- [x] 无邮箱 / 无 OAuth / 无手机验证
- [x] **禁止游客入座**；未登录可**观战**

### 托管与战绩

- [x] 行动时限默认建议 120s，开局房主可配
- [x] 超时 → 强制托管（AI 代打）
- [x] **有效托管** = 本局 AI 实际替该玩家执行过 ≥1 步
- [x] 仅点托管但 AI 未行动 → 不惩罚
- [x] 有效托管且己方胜 → 该玩家**不计 ranking 胜**
- [x] 有效托管且己方负 → 对方**正常计胜**

### 其它规则

- [x] 团队模式**允许**军阀拆队友
- [x] 拆队友：UI 明显警告 + 二次确认（服务端不禁拆）

### 部署 / 依赖（已有文档）

- [x] VPS + 一体 Node；DB 同机
- [x] Node 20；依赖基线见 DEPENDENCY_BASELINE
- [x] （可选）根 README / 本文与 DEV_DEPLOY 交叉链接已更新
- [x] 本地一键脚本说明：`scripts/README.md`

### P0 验收

- [x] 新贡献者只读 ROADMAP §P0 能回答模式/计分/账号/托管问题

### P0 明确不做

- 全量升级技术栈
- 娱乐模式详细规则设计
- Supabase / Vercel

## P1 — 账号最小集 + 入座门槛

> 目标：有帐号才能坐入玩家位；未登录只能观战。

### 切片 P1.1 — 服务端账号存储

- [x] 选定存储：SQLite（推荐 `better-sqlite3` 或等价）路径可配置
- [x] 表 `users`：`id`, `username`（唯一）, `password_hash`, `display_name`, `created_at`, `updated_at`
- [x] 注册 API：帐号名 + 密码 + 可选昵称（默认=帐号名）
- [x] 登录 API：签发会话（JWT 或 httpOnly cookie / token）
- [x] 密码哈希（argon2 或 bcrypt）
- [x] 环境变量：`DATABASE_PATH`、`JWT_SECRET`（或会话密钥）

**涉及（预期）：** `server/src/` 新增 `auth/`、`db/`；`.env.example`

**验收：**

- [x] 能注册、登录、错误密码拒绝
- [x] 密码非明文落库

### 切片 P1.2 — 客户端登录 UI

- [x] 登录 / 注册页或弹窗（中文文案）
- [x] 保存会话（token）；刷新后仍登录
- [x] 昵称修改入口（简单设置即可）
- [x] 未登录状态在 UI 上可区分

**涉及（预期）：** `client/src/` 路由、store、i18n

**验收：**

- [x] 中文界面完成注册→登录→改昵称流程

### 切片 P1.3 — 入座与观战

- [x] Socket 连接携带认证信息
- [x] `join room` 作为**玩家**：必须已登录；绑定 `userId`
- [x] 未登录：只能观战（或明确拒绝入座并提示登录）
- [x] 断线重连：优先 `userId + roomId`，弱化纯匿名 localStorage（可保留作辅助）
- [x] 大厅人数统计区分玩家 / 观战

**涉及（预期）：** `server/src/socket/server.ts`、`Room`、`Player`；`client` 进房逻辑

**验收：**

- [x] 未登录无法成为对局玩家
- [x] 已登录可入座并开始（人数足够时）
- [x] 观战不占玩家位（行为符合设计）

### P1 明确不做

- 邮箱验证、OAuth、排行榜页
- 3v3 规则改动（留给 P2）

## P2 — 6 人强制 3v3 + 竞技计分

> 目标：六人开局即团队竞技；终局按队伍总分判胜负/平局。

### 切片 P2.1 — 协议与模式字段（common）

- [x] `GameMode`: `competitive_team6` | `casual`（命名可微调，语义固定）
- [x] `TeamId`: `A` | `B`（或 0/1）
- [x] 客户端状态导出：每玩家 `team`；终局 `teamScores`、`matchResult`（胜/负/平）
- [x] 竞技固定 `completeCitySize = 8`

**涉及：** `common/src/index.ts`（及必要类型）

**验收：**

- [x] common build 通过；server/client 能编译引用新类型

### 切片 P2.2 — 开局与分队（server）

- [x] 6 名**玩家**开局 → 强制 `competitive_team6`，不可选个人混战
- [x] 非 6 人 → `casual`
- [x] 分队：开局时 `playerOrder` 的 0,2,4 = 队 A；1,3,5 = 队 B；整局固定
- [x] 大厅：6 人时隐藏/禁用「个人模式」类选项（若有）

**涉及：** `GameState.setupGame`、`BoardState`、`socket` `start game`、大厅 UI

**验收：**

- [x] 6 人开局状态中人人有正确 `team`
- [x] 无法以 6 人个人混战开局

### 切片 P2.3 — 建成加分与队伍结算

- [x] 跟踪「本局是否已有人建成」与「当前回合（角色行动轮次）内建成顺序」
- [x] 首个建成：**+8**；同回合内后续建成：**+6**
- [x] 个人分仍按原公式其它项计算，再叠加建成奖励
- [x] 终局：`teamScore` 汇总；比较两队 → 胜 / 负 / **平局**
- [x] `FINISHED` 状态推送含队伍分与结果

**涉及：** `GameState`（`finishTurnPhase` / `computeScores` 等）、`PlayerBoardState`

**验收：**

- [x] 单测或脚本/手动：模拟首完与同回次完，加分为 8 与 6
- [x] 队总分相等时结果为平局

### 切片 P2.4 — 客户端展示

- [x] 座位/列表显示队色或队名（A/B）
- [x] 终局面板：个人分 + 队伍总分 + 胜负/平局
- [x] 竞技局提示「3v3 竞技」

**涉及：** `LobbyScreen`、`GameScreen`、`PlayerScore` 等；`locale.zh.json` / `en`

**验收：**

- [x] 双浏览器能看清己方队伍与终局队伍结果

### P2 明确不做

- 军阀禁拆队友
- 战绩入库（P3）
- AI（P5）

## P3 — SQLite 战绩 + 竞技排名

> 目标：对局可查；仅竞技局进排名。

### 切片 P3.1 — 表结构

- [x] `matches`：模式、开始/结束时间、结果摘要、是否 ranked、是否含 AI 等
- [x] `match_players`：user_id、seat、team、个人分明细、是否有效托管、是否计 ranking 胜 等
- [x] 含 AI 或非 `competitive_team6` → `ranked = false`（娱乐）

**验收：**

- [x] 迁移/建表可重复执行；本地 DB 文件路径符合 DEV_DEPLOY

### 切片 P3.2 — 终局写入

- [x] 对局 `FINISHED` 时服务端写入 match + players
- [x] 游客/异常局策略明确（无 user 的座位跳过或标记）
- [x] 失败写入不拖垮进程（日志 + 可重试策略简单即可）

**验收：**

- [x] 打完一局竞技后 DB 有记录
- [x] 娱乐局有记录且 `ranked = false`

### 切片 P3.3 — 查询与 UI

- [x] API：我的战绩列表；简单排名（仅 ranked 胜场/场次等，指标保持简单）
- [x] 客户端：战绩页 / 简单排行
- [x] 显示昵称而非帐号名

**验收：**

- [x] 登录后能看到自己历史局
- [x] 排名不含娱乐局

### P3 明确不做

- 复杂赛季、ELO、回放录像
- 多实例分库

## P4 — 倒计时 + 有效托管 + 战绩惩罚

> 目标：行动限时；超时 AI 接管；有效托管影响 ranking 胜场统计。

### 切片 P4.1 — 行动倒计时

- [x] 开局设置：行动时限（秒），默认 120，建议范围 60–180（测试允许 10）
- [x] 当前行动玩家服务端计时；超时触发托管逻辑
- [x] 客户端显示倒计时

**验收：**

- [x] 将时限调至 10s 可稳定复现超时托管（大厅可选 10s）

### 切片 P4.2 — 托管状态

- [x] 玩家可手动请求托管 / 取消托管（socket `set autoplay`）
- [x] 超时强制进入托管
- [x] `had_effective_ai_control`：AI **成功执行至少一步** 后置 true
- [x] 仅进入托管但 AI 未行动 → 仍为 false

**验收：**

- [x] 手动托管后取消且无 AI 行动 → 无有效托管标记
- [x] AI 代打一步后 → 有效托管为 true（`scripts/test-p4-autoplay.js`）

### 切片 P4.3 — 战绩惩罚落地

- [x] 终局写库时：有效托管 + 己方胜 → 该玩家 `ranked_win_eligible = false`（不计 ranking 胜）
- [x] 有效托管 + 己方负 → 对方队伍胜场正常统计
- [x] 排名聚合使用 `ranked_win_eligible`

**验收：**

- [x] 用测试局验证胜/负两种惩罚语义（`scripts/test-p4-autoplay.js`）

### P4 明确不做

- 复杂反作弊
- 高难度 AI（P5）

## 推荐推进顺序（摘要）

```
P0（已定规格）
 → P1 账号与入座
 → P2 3v3 竞技规则（可与 P1 后半交错，但开局校验依赖登录人数）
 → P3 战绩落库与排名
 → P4 倒计时与托管惩罚（依赖 P3 字段）
 → P5 AI（可部分与 P4 并行，但有效托管依赖 AI 执行）
 → P6 UX
 → P7 上线
```

**建议每周只完成 1 个切片（P1.1 / P2.3 这类），** 适配 SuperGrok + Kilo 的额度与上下文。
